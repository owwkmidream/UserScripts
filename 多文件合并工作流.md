# 单一大文件脚本拆分与单文件发布通用指南

> 适用场景：历史上只有一个超大 `*.user.js`（或单函数巨型脚本）的项目，希望提升可维护性，但发布形态仍保持“单文件可安装”。

## 1. 目标与边界

- 目标一：开发形态从“单大文件”升级为“多模块源码”。
- 目标二：发布形态保持“单文件脚本”，不影响现有安装链接和使用方式。
- 目标三：先做结构迁移，再做功能迭代，避免一次性改动过大导致回归。

边界约束：
- 不在拆分阶段改业务行为。
- 不在拆分阶段改接口协议。
- 不在拆分阶段改用户可见交互（除非为了修复明显 bug）。

## 2. 推荐目录模板

```text
YourScript/
  package.json
  merge.config.js (可选)
  README.md
  src/
    meta.user.js
    index.js
    app.js
    constants.js
    state.js
    utils.js
    feature-a.js
    feature-b.js
    render.js
    styles.js
../YourScript.user.js
```

说明：
- `src/` 是唯一可维护源码。
- `../YourScript.user.js` 是唯一发布产物。
- 强制路径约束：发布产物必须位于 `package.json` 所在目录的上级目录。
  - 例：若 `package.json` 在 `YourScript/`，则产物路径应为 `../YourScript.user.js`。
- `meta.user.js` 单独维护，发布时拼接到产物头部。

### 2.1 合并工具建议（默认 `rolldown`）

- 建议把 `rolldown` 作为默认方案写进文档。
- 理由：在“多模块源码 -> 单文件发布产物”场景下，`rolldown` 通常有较好的构建性能与较低配置成本。
- 结论边界：不宣称“绝对最快”；如团队已有稳定链路，可继续使用现有工具。

默认实现要求（与具体工具无关）：
- 入口固定为 `src/index.js`
- 元数据头来自 `src/meta.user.js`
- 产物固定输出到 `package.json` 上级目录：`../*.user.js`
- 由自动化流程生成产物，禁止手改产物

`rolldown` 最小示例：

`package.json`

```json
{
  "type": "module",
  "scripts": {
    "build": "rolldown -c",
    "build:watch": "rolldown -c -w"
  },
  "devDependencies": {
    "rolldown": "^1.0.0-beta.0"
  }
}
```

`rolldown.config.mjs`

```js
import { readFileSync } from 'node:fs';
import { fileURLToPath } from 'node:url';
import { defineConfig } from 'rolldown';

const meta = readFileSync(new URL('./src/meta.user.js', import.meta.url), 'utf8').trimEnd();
const inputFile = fileURLToPath(new URL('./src/index.js', import.meta.url));
const outputFile = fileURLToPath(new URL('../YourScript.user.js', import.meta.url));

export default defineConfig({
  input: inputFile,
  treeshake: false,
  output: {
    file: outputFile,
    format: 'iife',
    name: 'UserScriptApp',
    sourcemap: false,
    banner: `${meta}\n`,
  },
});
```

## 3. 模块职责标准

- `meta.user.js`：Userscript 头信息（`@name/@grant/@require/@connect/@match`）。
- `index.js`：最薄入口，只做“样式注入 + 启动”。
- `app.js`：调度层，负责初始化、轮询、事件绑定和模块编排。
- `constants.js`：状态枚举、URL、DOM ID、时间常量等魔法值。
- `state.js`：跨模块共享状态，避免全局变量散落。
- `utils.js`：纯工具函数与请求封装。
- `feature-*.js`：按业务域拆分（任务、直播、数据、通知等）。
- `render.js`：DOM 渲染、卡片更新、提示层。
- `styles.js`：样式字符串与注入逻辑。

## 4. 拆分原则

- 单一职责：一个模块只做一类事。
- 依赖单向：入口层依赖业务层，业务层依赖基础层；避免反向依赖。
- 副作用收口：定时器、事件监听、DOM 挂载集中在 `app/render`。
- 状态集中：共享可变数据只放 `state.js`。
- 发布统一：所有发布都由自动化流程生成，禁止手改产物文件。

## 5. 标准迁移流程（SOP）

### 5.1 冻结基线

1. 复制当前单文件作为基线版本（可回滚）。
2. 记录现有入口、定时器、全局事件、网络请求、外部依赖。
3. 列出“不可变行为清单”（关键按钮、关键链路、关键提示）。

### 5.2 建立发布骨架

1. 新建 `src/` 与发布流程配置（或发布脚本命令）。
2. 把 Userscript 头提取到 `src/meta.user.js`。
3. 建立 `src/index.js` 与 `src/app.js` 最小可运行骨架。
4. 配置发布输出到历史产物路径，确保安装地址不变。
   - 建议固定为“项目根目录的上级”，即相对 `package.json` 使用 `../*.user.js`。

### 5.3 机械拆分（不改逻辑）

1. 提取 `constants.js`（URL、ID、枚举、时间配置）。
2. 提取 `utils.js`（纯函数、请求封装、格式化）。
3. 提取 `state.js`（原全局变量集中管理）。
4. 按业务域拆到 `feature-*.js`（先复制原逻辑，再局部整理）。
5. 把渲染相关集中到 `render.js`，与数据处理分离。

### 5.4 编排回归

1. 在 `app.js` 串起初始化、轮询、事件触发和主刷新流程。
2. 保证 `index.js` 只调用 `injectStyles()` 和 `start()`。
3. 逐步替换旧调用链，直到单文件旧实现可以完全删除。

### 5.5 产物发布

1. 通过自动化合并流程生成单文件脚本。
2. 校验产物包含完整 Userscript 头。
3. 校验产物输出路径、文件名、安装地址与历史一致。
   - 路径必须在 `package.json` 所在目录的上级目录。

## 6. 用户脚本场景专项规则

- `GM_*` 调用集中封装，避免散落。
- `@grant` 必须与实际调用一致，拆分后不要漏。
- `@connect` 必须覆盖所有请求域名，拆分后不要漏。
- `@require` 外链依赖需保持可访问，避免版本漂移导致运行差异。
- 依赖浏览器全局对象（如 `unsafeWindow`）时，统一放在边界层读取。

## 7. 验证清单

### 7.1 静态验证

- 所有模块 `import/export` 可解析。
- 无明显循环依赖。
- 无重复常量与硬编码 URL 扩散。
- 入口链路清晰：`index -> app -> feature/render`。

### 7.2 行为验证

- 面板或核心入口可正常显示。
- 主刷新链路可正常触发并更新 UI。
- 关键操作按钮行为一致（成功与失败提示都一致）。
- 异常路径可用（网络失败、权限不足、登录缺失、风控等）。

### 7.3 发布验证

- 发布产物可被脚本管理器安装。
- 元数据头完整。
- 脚本版本号、名称、匹配规则正确。

## 8. 风险控制与回滚

- 每次迁移只做一类结构变更，不叠加新需求。
- 保留“上一版单文件产物”以便快速回滚。
- 拆分阶段建议小步提交：`constants -> utils -> state -> feature -> render`。
- 任一步出现不可解释行为差异，先回到上一步，不硬推。

## 9. 常见反模式（避免）

- 把业务逻辑继续堆到 `app.js`，导致“新瓶装旧酒”。
- 在多个模块直接读写 DOM，渲染责任不清。
- 拆分后仍靠隐式全局变量通信。
- 手动修改发布产物，导致源码与发布文件长期不一致。
- 一次性重写全部逻辑，无法定位回归来源。

## 10. 快速启动清单

1. 建立 `src/` 与发布脚本/配置。
2. 提取 `meta.user.js`。
3. 建立 `index.js + app.js` 最小入口。
4. 把常量和工具先抽离。
5. 按业务域拆分模块。
6. 统一渲染层。
7. 自动化生成单文件发布产物。
8. 按清单静态验证 + 手动验证。

---

本指南是通用流程文档，适用于任意“单一大文件脚本 -> 多模块源码 + 单文件发布”的改造任务。
